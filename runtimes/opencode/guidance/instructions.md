# Citadel Harness — OpenCode Runtime

An agent orchestration system running on OpenCode. This repo IS the harness.

## What This Is

The harness is a collection of skills, state files, and orchestration patterns that enable
OpenCode to operate autonomously on any codebase. It provides:
- Intent routing via the `citadel` skill that dispatches tasks to the right tool
- Campaign persistence across sessions
- Parallel agent coordination with discovery sharing
- Quality enforcement through instruction-based policies

## Key Conventions

- Skills live in Citadel's `skills/` directory, linked into OpenCode via `~/.config/opencode/skill/`.
- Each skill is loaded via OpenCode's native `skill` tool: `use skill tool to load {skill-name}`.
- The `citadel` skill is the main router — equivalent to Citadel's `/do` command.
- Campaign state lives in `.planning/campaigns/` (per-project).
- Fleet session state lives in `.planning/fleet/` (per-project).
- Configuration lives in `.citadel/harness.json` (generated by the `setup` skill).
- Citadel root is at `~/.config/opencode/Citadel/`.

## Differences from Claude Code Runtime

- **No lifecycle hooks**: OpenCode doesn't support Claude Code's hook system. Safety policies
  (file protection, external action gating) are enforced through these instructions.
- **Skill invocation**: Use `skill` tool instead of slash commands. Load skills explicitly.
- **No plugin system**: Citadel integrates via symlinks and instructions, not a plugin adapter.

## Safety Policies (Instruction-Enforced)

Since OpenCode doesn't have runtime hooks, enforce these policies manually:

### File Protection
- NEVER edit or overwrite these files directly: `.env`, `.env.*`, `credentials.json`,
  `*.pem`, `*.key`, `secrets.json`, `harness.json`, `settings.json`
- If a task requires modifying a protected file, ask for explicit confirmation first.

### External Action Gating
- BEFORE any `git push`, `gh pr create`, or similar external action, confirm with the user.
- Default behavior: always ask before external actions.
- If user says "auto-allow" for a specific action type, respect that for the session.

### Circuit Breaker
- If 3 consecutive tool calls fail with the same error pattern, STOP and report the issue.
- Do not retry the same approach more than twice. Try an alternative or ask for guidance.

### Quality Gate
- After completing implementation work, run the project's typecheck and test commands.
- Report results. Do not claim success if tests fail.

## Skill Routing

The `citadel` skill is the main entry point. It replaces `/do` from the Claude Code runtime.
When you need to route user intent, load the `citadel` skill and follow its routing protocol.

Key routing targets (loaded via skill tool):
- `citadel-review` → 5-pass structured code review
- `citadel-marshal` → Single-session orchestrator
- `citadel-archon` → Multi-session campaigns
- `citadel-fleet` → Parallel agent coordination
- `citadel-setup` → First-run configuration
- `citadel-research` → Structured investigation
- `citadel-refactor` → Safe multi-file refactoring
- `citadel-test-gen` → Test generation
- `citadel-doc-gen` → Documentation generation
